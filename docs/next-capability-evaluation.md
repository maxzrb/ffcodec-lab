# 后续能力评估（schema v4 完成后）

评估日期：2026-07-12

本评估完成既定计划的最后一步：在真实色彩处理和高级质量第二批稳定后，判断剪辑区间、元数据、章节、附件与自由滤镜排序的实施顺序。本文件只确定边界与优先级，不把尚未实现的能力伪装成已支持。

| 能力 | 用户价值 | 架构影响 | 主要风险 | 决策 |
|---|---:|---:|---|---|
| 剪辑区间 | 高 | 中 | `-ss/-to/-t` 放在输入前或输出前语义不同；copy 模式关键帧不精确 | 下一批优先，先做受控“快速剪辑 / 精确剪辑”，不开放任意组合 |
| 自定义元数据 | 高 | 低 | 键名、流作用域和清除策略需要受控；分享载荷不能带本地隐私 | 下一批优先，先支持全局与标题/作者等常用字段 |
| 章节 | 中 | 高 | ffmetadata 临时文件、外部路径和浏览器纯前端无法生成本地文件 | 暂缓；先支持保留/移除输入章节，再评估外部章节文件 |
| 附件 | 中 | 高 | 仅 MKV 等容器适用；MIME、额外输入和 `-attach` 路径涉及隐私与兼容性 | 暂缓到容器专项，必须先建立附件能力表 |
| 自由滤镜排序 | 中 | 很高 | 会破坏当前已验证的固定顺序、字幕末端约束和唯一 `-vf` 不变量 | 不直接开放自由拖拽；后续只考虑带依赖约束的有限排序 |

## 下一批建议范围

1. `ProjectConfig` 下一版本只加入剪辑与受控元数据，旧配置迁移后完全禁用，不改变命令。
2. 快速剪辑使用输入前 `-ss`，精确剪辑使用输出侧时间控制并明确需要解码；copy 模式显示关键帧精度提示。
3. 元数据采用键值数组但限制危险/空键，命令统一进入 `METADATA` 阶段。
4. 章节、附件和有限滤镜排序分别立项，不与剪辑/元数据共用一次 schema 升级。

## 进入实施前置条件

- schema v1–v4、预设与分享往返全部通过。
- FFmpeg 实机矩阵持续覆盖 zscale、tonemap、基础高级质量和单一滤镜链。
- 桌面/移动端工作台回归通过，字幕栏切换不再造成页面横向位移。
- catalog audit 保持 0 errors / 0 warnings。
